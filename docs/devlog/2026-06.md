## 2026-06-08 — URL-backed Board/Graph search routes (#63)

**Result** — PR [#63](https://github.com/sympoies/symphony-board/pull/63)
(squash-merged):
- The visible search bar is now **URL-backed**: `?q=` in the hash is the source
  of truth, and an absent `q` clears search instead of preserving hidden React
  state. Board/Graph/Settings tab links carry the current `q`, so the URL and
  the active filters stay aligned while navigating.
- Added a shared `buildHashRoute` next to `parseHashRoute`; board card
  focus-links, page tabs, and search input updates now use the same hash route
  shape. `graphFocusHref(item)` still opens `#/graph?focus=<id>&q=<repo #iid>`,
  but plain `#/graph` means no search filter.
- Fixes the Board -> Graph -> Board -> Graph drift where a deep-link token like
  `sympoies/symphony-board #56` stayed in the search bar / filters even though
  the URL no longer carried `q`.

**Why / context**
- #50 made board -> graph deep-links fast by seeding `?q=<repo #iid>`, but the
  later plain tab routes (`#/`, `#/graph`) did not encode search. The old
  `applyRouteSearch` intentionally left existing search alone when `q` was
  absent, which made tab navigation depend on hidden component state instead of
  the route the user could inspect, reload, or share.
- Search affects both the Board and the Graph, so it needs one visible contract:
  if a filter is active, it appears as `q`; if no `q` appears, no search filter
  is active. This preserves the useful #50 deep-link behavior while removing the
  confusing stale-filter case.

**Evidence**
- Model tests cover `buildHashRoute`, graph focus-link round-tripping, and
  absent-`q` clearing (`applyRouteSearch mirrors the route q so search never
  hides outside the URL`).
- Local validation before merge: `pnpm run typecheck`; `pnpm test` (56);
  `pnpm --filter @symphony-board/ui run build`; `pnpm --filter
  @symphony-board/ui run smoke`, including the deep-link seeded-search
  assertion. A focused headless flow also verified: board graph link -> Board
  tab keeps `#/?q=...`; Graph tab keeps `#/graph?q=...`; clearing search removes
  `q`; returning to Board has no hidden search.
- GitHub PR checks for #63 were green (`check`, `ui`, `coverage`, and CodeQL),
  and the delivery review outcome comment recorded testing + maintainability
  pass.

**Links** — PR [#63](https://github.com/sympoies/symphony-board/pull/63)
(`6c0fd2d`); builds on #50 (`?q=` graph narrowing) and #57 (clearing a
deep-link search resets the graph window).

## 2026-06-08 — Board Closed-column sort fix + per-column caps + graph window reset (#57)

**Result** — PR [#57](https://github.com/sympoies/symphony-board/pull/57) (open, checks green):
- **Fixed the board Closed/Trailing ordering.** `listLiveItems` now orders by
  `COALESCE(closed_at, merged_at, updated_at, created_at) DESC, item_id DESC`
  (was `(closed_at IS NULL) DESC, …`, which omitted `merged_at`). + a regression
  test in `test/db.test.ts`.
- **Per-column render cap of 100**, with the true total kept in the header and a
  `+N more` footer, on the Closed/Trailing status columns + every Spotlight lane
  (was a hard 30-item slice inside `spotlight()`; the lane now returns its full
  sorted list and the view owns the cap).
- **Graph time-window resets to the 90-day default** when a board→graph
  deep-link's narrowing `?q=` search is cleared (it was leaving the all-time
  window in place and dumping the entire history); a manual window pick opts out.

**Why / context**
- The Closed column always led with GitLab (Gamania) merged MRs because GitLab
  carries `merged_at`, **not** `closed_at`, for a merged MR — so every GitLab
  merge has `closed_at = NULL`, and the `(closed_at IS NULL) DESC` primary key
  floated all of them ahead of dated items. The UI partitions the contract's
  item order into columns **without re-sorting**, so the contract order *is* the
  column order — the fix had to live in `listLiveItems`, not the view. Not a
  normalize bug: the `closed_at`/`merged_at` split is intentional; the ordering
  just has to treat `merged_at` as a resolution time too.
- The cap lives in the **view**, not the query/contract. The contract is a
  shared surface (board + graph + stats + settings), so a `LIMIT` in
  `listLiveItems` would drop graph nodes and skew `byState` counts. The contract
  still ships every item — the real bottleneck is DOM render (1500+ cards), not
  transfer, so capping *render* is the right lever.

**Evidence**
- Live DB: GitHub merged 1214/1214 carry `closed_at`; GitLab (Gamania) merged
  75/75 have `closed_at = NULL` (only `merged_at`) — the exact rows the old key
  mis-floated. Contract is 1527 items / 1.93 MB, **gzip ~131 KB**.
- Post-fix `pnpm run emit` leads the Closed column with the most-recently-
  resolved items (GitHub 2026-06-07) instead of the Gamania cluster.
- Root `pnpm run typecheck && pnpm test`: clean + 51 pass (incl. the new
  ordering test). `packages/ui`: typecheck clean + 26 pass.

**Links**
- PR [#57](https://github.com/sympoies/symphony-board/pull/57)
- Follow-up issue [#56](https://github.com/sympoies/symphony-board/issues/56) —
  windowed-contract redesign (full aggregates + windowed item set + contract
  version bump), deliberately deferred.

## 2026-06-08 — Repo/source highlight colors + source-level hide (#49)

**Result** — #49 squash-merged (`7269c8f`):
- Optional highlight colors, config-defined: a source-level `color`, and per-repo
  via a project entry's object form `{ "path", "color" }` in `config/sources.json`
  (most repos stay bare strings). They ride the contract on an **additive minor
  bump 1.0.0 → 1.1.0**: `SourceDTO.color` + a new sparse top-level `repos[]`, both
  optional/non-required, so old readers are untouched (the UI gates on major only).
- The UI renders a highlighted repo/source as a **left color bar** on board +
  graph side-list cards and an **outer ring** on canvas nodes (the node's left edge
  already encodes state). Effective color resolves **override → repo → source →
  none**.
- Settings gains a **per-repo color override** (localStorage, with reset) and a
  **source-level hide** — an independent visibility layer from the per-repo hide,
  so toggling a source never clears the remembered per-repo choices.

**Why / context**
- Colors are display metadata: read from config at **emit** time and injected into
  `buildContract` (kept a pure mapping), never stored in the DB — so a color change
  needs only a re-emit (no re-fetch) and can't corrupt the canonical store. The
  3-layer separation holds: the contract gains the field; LAYER 2 doesn't.
- Resolution lives in the UI (where the override also lives); the config layers
  ship raw on the contract rather than being pre-resolved server-side. The repo
  join is `(source_id, project_path)` — GitLab stores the queried path verbatim,
  GitHub the canonical `nameWithOwner`, so the lookup carries a case-insensitive
  fallback. The card bar is a `::before` (not a border/box-shadow override) so it
  survives the hover / active / `:target` states, which already own those channels.
- **Pre-merge gate (4 parallel lenses): contract/api, maintainability, and testing
  clean; security/red-team surfaced a CSS-sink hardening.** A color reaches inline
  CSS — the `;`-breakout-into-a-new-declaration vector isn't real (the CSSOM rejects
  it), but a `url(…)` value via `background: var(--repo-color)` was a low residual
  reachable through a hand-edited localStorage override or a malicious/old contract.
  Added an `isHexColor` guard at the chokepoints (`buildColorIndex`,
  `loadColorOverrides`, the Settings swatch) so only `#rgb`/`#rrggbb` ever reaches
  the DOM (`0a731a3`).
- **Built over concurrent UI work** (#46 board→graph deep-link, #48
  updated-timestamp-first): a 5-file merge conflict (`App`, `ItemCard`, `FullBoard`,
  `GraphPage`, the render smoke) resolved to keep BOTH feature sets — `colorOf`
  threads alongside `linkedIds` / `focusRef`, and the card takes `accentColor` +
  `linked`.

## 2026-06-08 — Root-caused & killed the raw-NUL-in-source bug class (#41)

**Result** — #41 squash-merged (`7310de8`), closing the follow-up flagged in the
#32 entry below:
- Three composite Map-key builders joined fields with a NUL (`\0`) separator, and
  some were committed with a **literal `0x00` byte** instead of the source escape:
  `src/model/edges.ts` `keyOf` (2), `packages/ui/src/model.ts` `relatedItems` (1);
  `src/sources/gitlab.ts` `indexKey` had been the `\0`-escape form (fixed in #33).
- All three now build their key with `JSON.stringify([...])` — the same
  collision-proof convention as the UI model's `repoKey` — so there is **no
  separator byte to mis-save** at all. (`relatedItems` was independently rewritten
  in #40 to key by `ref`, so that site landed NUL-free via the rebase.)
- New `test/no-nul-bytes.test.ts` enumerates `git ls-files` and fails on any
  `0x00` byte; it runs in CI's `check` job and the pre-push hook, and was
  negative-tested (fails + names the file when a NUL is staged).

**Why / root cause**
- **Authoring-time, not a formatter.** The repo has **no formatter** (no
  prettier/biome/eslint; lefthook only runs typecheck + test), so nothing
  rewrites files on commit — whatever agent wrote the file emitted a raw NUL from
  an intended `\0`, and it was committed verbatim. Claude Code's own Edit/Write
  round-trip `\0` correctly (the #33 fix stayed clean), so the culprit is a
  different write path; commits are all signed `graysurf`, so history can't name
  the tool.
- **Why it survived for so long:** a NUL inside a JS/TS string is *valid* — `tsc`,
  the runtime, and behaviour tests (which assert results, not key bytes) never
  flag it. The only symptoms are `file`/`grep` seeing the file as binary and a
  `git diff` "Bin …" when the byte is within git's ~8 KB binary-sniff window.

**Evidence** — `pnpm run typecheck` + `pnpm test` (40 pass incl. the new guard),
UI build + `… run test` (14 pass), and a full-tree NUL scan → zero. The fix is
behaviour-preserving: every changed key is an in-memory dedup Map key, never
persisted to the DB or contract.
