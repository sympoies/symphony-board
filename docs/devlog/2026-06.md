## 2026-06-07 — Graph navigation: side list, click-to-focus, node metadata (#24)

**Result** — #24 shipped via squash PR #27 (`7ad7a7e`); the deployed `web`
sidecar was rebuilt to serve it:
- A searchable, demand-sorted **side list** beside the graph canvas
  (`[list | canvas]`); clicking an entry pans + zooms onto the node via React
  Flow `fitView({ nodes: [id], maxZoom: 1.5, duration: 600 })`. The default view
  stays the full-graph fit.
- Node cards now carry **created / updated (relative) + demand**; `created_at` /
  `updated_at` were added to `GraphNode` and carried through `buildGraph` (null
  for untracked endpoints).
- Redeployed the `web` image from `origin/main` HEAD (`e24159b`, which also
  picked up the concurrently-merged #26 Settings filter and #28 source mark).

**Why / context**
- **The side list must drive the canvas camera from *outside* `<ReactFlow>`.**
  RF's `useReactFlow().fitView` only works inside a provider, so the list and the
  canvas now share one explicit `<ReactFlowProvider>`; the canvas keeps its
  `key`-based remount (drag/filter reset) underneath it. `fitView({ nodes: [id] })`
  reads the node's measured box, so a readable `maxZoom` + a `duration` give the
  zoom-to-node the dense default view couldn't.
- **No contract change.** `created_at` / `updated_at` / `demand` are existing
  nullable contract fields — this is presentation + navigation only, and stayed
  inside the lazy `GraphPage` chunk.
- **Deploy without touching the active checkout.** The shared main checkout sat
  on an unrelated WIP branch, and `data/` is bind-mounted relative to the compose
  file, so a plain `up --build` would have shipped the WIP, not `main`. Built the
  `web` image with `git archive origin/main | docker build -f docker/ui.Dockerfile -`
  (no branch switch, no worktree `data/`), then `up -d --no-build --force-recreate
  web` reused the main checkout's mounts; the `board` daemon was left running and
  healthy.

**Evidence**
- PR #27 squash-merged, issue #24 auto-closed (`Closes #24`); CI `check` + `ui` +
  CodeQL green.
- `pnpm run typecheck` + `pnpm -r test` (41) + UI `build` + `render-smoke` (0
  console errors) + `verify-board.mjs`, plus a CDP focus probe: side list = 9
  items, filter → empty state, **viewport scale 0.31 → 1.5 on a list click**, 8
  meta rows / 9 nodes (the untracked node correctly has none).
- Served `:8080` after redeploy: `/` and `/contract.json` → 200; the served
  `GraphPage` chunk contains `graph-list` / `rf-node-meta` / `filter items` /

## 2026-06-07 — Deploy pipeline (daemon → sidecar), GitLab mentions, graph polish

**Result** — issues #11–#15 closed via five squash PRs (#16–#20):

  Spotlight lane definitions into a declarative `packages/ui/src/spotlight.config.ts`.
- **#12 daemon as source** (#17): a `board` healthcheck reports healthy only
  while `data/contract.json` keeps being refreshed; documented that the daemon is
  the sole writer of the served contract.
- **#11 deploy** (#18): a `web` nginx sidecar (multi-stage Vite build, in
  `docker/ui.Dockerfile`) serves the built board plus the daemon-emitted contract
  at `http://localhost:8080`, gated on the board healthcheck — no manual
  `vite preview`.
- **#13 GitLab cross-refs** (#19): parse GitLab **system notes** for `mentions`
  ("mentioned in #N / !N") and `relates` ("marked as related to #N"); bumped
  `normalizerVersion` to `gitlab/2`.
- **#15 graph polish** (#20): node size by demand, neighbour highlight + edge-type
  labels on hover, and a mention-direction filter.

**Why / context**
- **GitLab GraphQL has no `relatedIssues` and no `CrossReferencedEvent`** — only
  `relatedMergeRequests` (the existing `closes` source). Live introspection
  confirmed this, so the chosen "both relates + mentions" is realized by parsing
  human-readable **system notes** instead (locale/version dependent, brittle by
  nature). Notes are fetched **per item** (an N+1, like `relatedMergeRequests`):
  inlining `notes(first:N)` in the bulk page would exceed GitLab's query-complexity
  limit. A mention that duplicates a `closes` edge (the closing MR) is skipped,
  mirroring GitHub's `willCloseTarget`; the symmetric `relates` link is
  canonicalized so reconcile merges the two reports into one edge. `normalize`
  stays pure — `fetch` resolves iid → global id (via the sweep's items) and
  attaches the endpoints to the payload.
- **The UI build can't run the workspace `prepare` in Docker** — lefthook's
  `install` needs a git repo the image hasn't got. `pnpm install --ignore-scripts
  && pnpm rebuild esbuild` skips `prepare` while still building Vite's one
  allowlisted dependency.
- **Sole-writer holds end to end**: the sidecar `depends_on` the board
  healthcheck (`condition: service_healthy`), so the reader never serves before
  the writer has produced a contract.

**Evidence**
- All five PRs squash-merged; CI (`ci` = check + ui, plus CodeQL) green on every
  commit (`948841b`, `6c48b59`, `4ff5cf9`, `65b649b`, `2ee818d`).
- `pnpm run typecheck` + `pnpm test` (35/35) + UI `build` / `test` (7/7) / `smoke`
  (17/17, 0 console errors) on the relevant branches.
- Daemon stood up against the fixture project set (3 GitHub repos + GitLab
  fixture): a full sweep emitted contract `1.0.0`, **333 items / 520 edges**;
  GitLab edges now `{closes:4, relates:1, mentions:1}`. The disappearance rule was
  observed live — a GitLab full+complete sweep soft-deleted 10 re-seeded-away

## 2026-06-07 — Contract pipeline hardening + the UI lands

**Result** — one branch (`feat/contract-pipeline-hardening`) closed every
remaining v1 deferral except raw history, and shipped the first UI:
- **Contract validator** (`9325123`): a dependency-free JSON-Schema-subset
  validator; `emit` now refuses to write an invalid contract. No ajv — keeps the
  backend's zero-third-party-dep posture.
- **Edge soft-delete** (`6b0b3dd`): a full+complete sweep tombstones unseen
  intra-source edges, symmetric to items.
- **Incremental loop cadence** (`f6bf2f7`): the daemon runs `--incremental` most
  iterations and a full sweep every `FULL_EVERY` (default 12 ≈ hourly), so the
  soft-delete stays correct while routine runs stay cheap.
- **`@symphony-board/contract`** (`5636dc2`): LAYER 3 extracted into a workspace
  package (pnpm workspace); the three-layer boundary is now structural.
- **`packages/ui`** (`15a84a1`): a read-only board (Vite + React) consuming the
  contract — source health, filters/grouping, label chips, and the issue↔PR/MR
  relationship view by lifecycle.

**Why / context**
- **The contract package is type-only at runtime — on purpose.** Node refuses to
  strip types from files under `node_modules`, so a runtime *value* imported from
  a workspace package would have forced a build step onto the backend. Keeping
  the producer constants (`CONTRACT_VERSION`, `GENERATOR`) and the validator on
  the producer side, and exporting only types + the JSON schema from the package,
  means the backend's `import type`s erase and it never resolves the package at
  runtime — so it still ships with no build and no `node_modules` in the Docker
  image. Proven by running build+validate with the workspace link removed.
- **The UI is the deliberate home for a build step.** DESIGN anticipated "heavy
  deps + a build step" for the UI; Vite/esbuild is exactly that. esbuild's
  build-script is the single trusted exception in the workspace `allowBuilds`
  allowlist, scoped to the UI's tooling — the backend stays minimal-trust.
- **The engine is a forwarder; the source owns the watermark logic.** A unit test
  pinned this (full → source nulls `since`; incremental → stops at the first
  record older than the watermark), plus a fake-Source engine test for the
  "incremental/partial never tombstone" gate.

**Evidence**
- `pnpm run typecheck` clean (backend + contract package); `pnpm test` 17 → **33**
  pass (added contract-validation, edge soft-delete, sync-engine gating, and the
  GitHub incremental-filter tests).
- UI: `pnpm --filter @symphony-board/ui run build` → `vite build` OK (~50 kB gzip
  JS); model tests 5/5; `vite preview` serves `index.html` + the co-located
  `contract.json` (200/200). The bundled `public/contract.json` sample validates
  clean against the schema via the backend validator.
- `pnpm install --frozen-lockfile` green (CI parity) after the workspace + UI deps.

## 2026-06-02 — Dev tooling: lefthook pre-push + .agents dispatchers

**Result**
- Adopted `setup-project` (`partial → adopted`): `.agents/scripts/{pre-pr,
  bootstrap,deploy}.sh` dispatch targets — pre-pr is the `typecheck && test` CI
  gate, bootstrap a frozen install, deploy the docker loop-daemon bring-up.
  Release skipped (private repo, no publish flow).
- Added a lefthook **pre-push** hook (typecheck + test in parallel, mirroring
  CI), auto-wired on `pnpm install` via a `prepare: lefthook install` script —
  fresh clones get hooks with no manual step. Commit `4a0c1fb`.

**Why / context**
- **pnpm 11 build-script gotcha (the keeper).** lefthook ships a postinstall;
  pnpm 11 blocks unapproved build scripts *and fails the install* (`exit 1`,
  via the pre-run deps-status check) — this would have broken CI's
  `pnpm install --frozen-lockfile`. The approval/ignore config has **moved to
  `pnpm-workspace.yaml`** (`allowBuilds: { <dep>: true|false }`); the
  package.json `pnpm` field (incl. `ignoredBuiltDependencies`) is **no longer
  honored** in 11.4.0 — verified by a fresh reinstall still erroring with the
  field set. Fix: `allowBuilds: { lefthook: false }` (acknowledge, don't run the
  dep's postinstall) + wire hooks via our own `prepare` (keeps the repo's
  minimal-trust, no-dependency-build-scripts posture).
- Chose **pre-push** over pre-commit (whole-project gate, no per-commit
  friction); **no commit-msg hook** (commit format is owned by the commit flow);
  the live `--dry-run` smoke is deliberately **not** gated (needs a token /
  possibly VPN-bound GitLab) — same reason CI skips it.

**Evidence**
- `rm -rf node_modules && pnpm install --frozen-lockfile` (CI parity) → `exit 0`,
  `prepare` auto-wires `pre-push`. Before the `allowBuilds` fix the same command
  was `exit 1`.
- `.agents/scripts/pre-pr.sh` and `lefthook run pre-push --all-files` →
  typecheck ✔, **17 tests pass**, `exit 0`.
- Commit `4a0c1fb` signed (key `…A470DF`), 8 files, on `main`.
