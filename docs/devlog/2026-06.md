## 2026-06-13 — URL-back the board/graph/repo-analytics facet lens (full single track)

**Result**
- The board / graph / repo-analytics item-facet lens (source / state / kind) is
  now route-backed via new `isource` / `istate` / `ikind` params, owned by
  `nav.ts` (`itemFacets` / `toggleItemFacet` / `itemFacetFields`). The shared
  React `filters` state is retired down to just `search`; `toggle()` is gone.
- One lens is shared across the three item pages and carried across every tab hop
  (`nav.tabHref` now threads it), matching the old sticky React-state behaviour —
  but it is now shareable, reloadable, and steps through browser history.
- Kept deliberately separate from the Activity feed's own
  `source/repo/kind/action` facets (distinct route fields), so the two never
  bleed when both ride a URL.

**Why / context**
- After the previous entry, Activity was route-backed but board/graph/repo-
  analytics still drove their chips from React state — a dual track. Their
  selections were not shareable, not deep-linkable, lost on reload, and absent
  from history; `Controls` and `filters` were also a single object shared
  board↔graph. This converges everything onto one mechanism (route is the truth).
- Decision (with the user): board/graph/repo-analytics SHARE one facet param set
  and CARRY it across tabs; Activity keeps its own facets. Repo Analytics had to
  move too — it shares the same `Controls`/`filters`, so leaving it on React
  state would have left a dead chip path.

**Result**

  single read-only session semantics only — no pooling/replica routing —

  `db_path` is meaningless under Postgres, where the precheck would have
  turned "store not migrated yet" into a wrong 404; pg fails loudly at open
  instead (locked by a test).
- postgres.js types `default_transaction_read_only` as a boolean connection
  parameter (not `"on"`); the live e2e is what proves the GUC actually pins

## 2026-06-11 — Writer lease + PgStore: the Postgres swap, proven live

**Result**
- The Store seam took its swap: `PgStore` (`src/db/postgres.ts`, postgres.js,
  `schema/postgres/0001–0003`) passes the same conformance suite as SQLite and
  is config-selected via `db_url_env` (env-var NAME holding a `postgres://`
  URL; the pg driver loads lazily so SQLite deployments never resolve the
  dependency). SQLite remains the production store.
- #164 landed in both halves: `Store.acquireWriterLease()` try-lock, the
  runner skipping wholesale on denial (`status: "skipped"`, surfaced to the
  UI); SQLite enforces it with `BEGIN EXCLUSIVE` on a sibling `<db>-lease`
  lock database, Postgres with `pg_try_advisory_lock` on a dedicated
  per-acquire session — both auto-release when the holder dies, no TTL
  machinery. Watermark upserts on pg are GREATEST-monotonic.
- New gate: `pnpm run test:pg-e2e` (and a CI `pg` job) composes a throwaway
  tmpfs Postgres and runs the conformance suite on both drivers plus a live
  e2e from zero — migrate, full sync, incremental, disappearance sweep,
  validated contract emit, refused second writer, `pg_terminate_backend`
  crash auto-release. `pnpm test` stays Docker-free.

**Why / context**
- The seam (#162) was built so a Postgres swap would be cheap; doing the swap
  now — with a live e2e — is what proves the conformance suite actually IS
  the swap guarantee rather than a hope. The lease moved "exactly one sync
  writer" from deployment topology into the database before a network-reachable
  store widens the second-writer hole to anyone holding the URL.
- Validation earned its keep twice. (1) The first full gate run hung: querying
  a crashed reserved connection trips an *uncatchable* async TypeError inside
  postgres.js — the lease became a dedicated session whose `end()` IS the
  release, `max_lifetime: null` pinned. (2) The one-time real-config sync into
  a throwaway pg failed on gitlab.com: `target_iid` 3430406968 exceeds int4,
  which SQLite's 64-bit INTEGER had silently absorbed forever. Provider
  numbers are now BIGINT + a 64-bit round-trip conformance case binds every
  future driver.

**Evidence**
- `pnpm run test:pg-e2e` 46/46 (19 conformance cases × 2 drivers + 7-step live
  e2e + parent), clean process exit; `pnpm test` 172/172 Docker-free;
  coverage 86.85% ≥ 85% floor; live `--dry-run` sync exit 0.
- Real-config full sync into throwaway pg, all three sources `status=ok`

**Result**

  the Compose network. Dry-run keeps `POST` honest — it opens no write
  transaction and emits nothing, so connectivity tests never present unwritten
  state as fresh data. See `docs/DESIGN.md` › "UI-Triggered Sync Control Plane".

## 2026-06-09 — Canonical actor identity for Repo Analytics (contract 2.3.0)

**Result**

  the raw display string, so one human is one row. Each row gained `actor_key`,
  `display_name`, and optional `aliases`; `actor` stays as a back-compat display
  field. Additive minor bump `2.2.1 → 2.3.0`.
- New `deriveActorKey` helper (`src/model/actor.ts`), additive DB migration
  `0003` (`activity.actor_key`, backfilled on upsert), and source normalizers
  that populate the key — including #93's review/approval activities.

**Why / context**
- Key precedence is **provider username → hashed commit email → normalized
  name**, deliberately *not* the issue's suggested email-first ordering: a
  GitHub user's issues carry a login but their commits carry an email, so
  email-first would re-split the same human — the exact duplication we were
  fixing. Username-first merges account-linked commits with issues; email is the
  fallback that collapses account-less commit author-name variants. The email is
  SHA-256-hashed, so a raw address never enters the canonical store or the
  contract.

  `Wayne XU(許恒彬)`/`Wayne` and `Terry LIN 林品澄`/`Terry LIN` collapsed into one
  email-keyed identity with aliases. The username and email facets of one GitLab
  human remain separate rows (events carry a username, commits carry an email,
  with no provider cross-link) — a documented follow-up, not the bug this fixed.

## 2026-06-08 — Root-caused & killed the raw-NUL-in-source bug class (#41)

**Result** — #41 squash-merged (`7310de8`), closing the follow-up flagged in the
#32 entry below:
- Three composite Map-key builders joined fields with a NUL (`\0`) separator, and
  some were committed with a **literal `0x00` byte** instead of the source escape:
  `src/model/edges.ts` `keyOf` (2), `packages/ui/src/model.ts` `relatedItems` (1);
  `src/sources/gitlab.ts` `indexKey` had been the `\0`-escape form (fixed in #33).

  separator byte to mis-save** at all. (`relatedItems` was independently rewritten

  `0x00` byte; it runs in CI's `check` job and the pre-push hook, and was
  negative-tested (fails + names the file when a NUL is staged).

- **Authoring-time, not a formatter.** The repo has **no formatter** (no
  prettier/biome/eslint; lefthook only runs typecheck + test), so nothing
  rewrites files on commit — whatever agent wrote the file emitted a raw NUL from

  different write path; commits are all signed `graysurf`, so history can't name
  the tool.

  the runtime, and behaviour tests (which assert results, not key bytes) never
  flag it. The only symptoms are `file`/`grep` seeing the file as binary and a
  `git diff` "Bin …" when the byte is within git's ~8 KB binary-sniff window.

## 2026-06-07 — Settings page: client-side per-repo display filter (#23)

- The choice persists in `localStorage` and is applied as a **global pre-filter
  before** the transient facet toggles: it narrows items **and** their edges into

**Why / context**
- **Display-only, client-side — deliberately NOT editing the daemon's sync
  config.** Changing what the daemon syncs would need a write API behind the
  static nginx sidecar and break the read-only-consumer architecture; the daemon
  stays the sole writer and keeps syncing every source. No backend, no contract
  bump.

  `source`/`state`/`kind` toggles and is the only view state that persists.
  Statuses are derived over the post-pre-filter `visibleEnv`, so a hidden repo's
  edges stop influencing Trailing / In-Progress everywhere — not just hidden from
  the column placement.
- **Store the HIDDEN set, not the visible set.** Tracking hidden keys keeps
  "everything visible" the default as new repos appear in later syncs.
- **`repoKey` is a JSON tuple** (pre-merge review repair, squashed in): a
  collision-proof composite key with no separator assumption — it doubles as the
  `localStorage` unit.
