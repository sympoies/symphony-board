## 2026-06-25 — Non-root containers (board + web), shipped in v1.9.6

**Result**
- Both Docker images now run **non-root**, resolving the two Trivy `DS-0002`
  (HIGH, "image runs as root") code-scanning alerts: the backend/board image as
  the built-in `node` user (uid 1000), and the web image switched from
  `nginx:alpine` to `nginxinc/nginx-unprivileged` (uid 101), listening on
  **8080** instead of 80. PR #472 (squash `b80a29b`), released as **v1.9.6** and
  deployed to the Postgres compose deployment.
- The container-port change is host-port-transparent: the compose files map the
  unchanged host web port to container 8080 (so any HTTPS front is unaffected).

**Why / context**
- The new SCA workflow (#459) surfaced the run-as-root HIGHs; SAST was already
  covered by CodeQL default setup, so the gap was purely the container privilege
  posture.
- Deploying non-root onto an EXISTING deployment is not free: the named volumes
  (`contract-data`, `live-data`, `live-telegram-data`) were root-owned, and
  Docker never re-chowns a populated volume — so the non-root containers cannot
  write them. The cutover needs a one-time `chown -R 1000:1000` of those volumes,
  done with the writers stopped so the live SQLite db is not re-rooted
  mid-migration. A fresh deploy needs no chown; `postgres-data` is left alone
  (the postgres image owns its own uid).
- Two non-obvious CI interactions, both fixed: the `pg_compose` smoke forces the
  config-control plane on and tests a config WRITE (`PUT /api/config`), which the
  non-root board cannot do unless its (throwaway) config dir is writable — the
  default deploy mounts config read-only and does not enable config-control, so
  it is unaffected. And the publish-image board smoke bind-mounts a host data dir
  that must be made uid-1000-writable for the `--dry-run`. The nginx/gzip path
  was never the problem (a debugging red herring): `gzip_static` serves fine
  non-root given world-readable files in a traversable dir — the real CI failure
  was the config-control write, only visible after adding a failure log dump to
  the smoke.

**Evidence**
- `pnpm run test:pg-compose` (full Postgres stack — postgres/board/api/web/live
  all healthy, valid contract served) green locally and in CI after the
  config-dir fix; root-caused on a real-Linux daemon (a Mac-local Docker VM masks
  the volume ownership enforcement).
- Post-deploy verified live: containers on 1.9.6, board = uid 1000 / web = uid
  101, `emit wrote … -> data/contract.json` succeeds as non-root, and the served
  `/contract.json` advanced to `generator: symphony-board/1.9.6`.
- A security-lens review found no blocking issues (privilege drop holds in both
  images; `chmod 0777` confined to CI-throwaway dirs, never a published layer).

**Links**
- PR #472 (non-root hardening, squash `b80a29b`); release **v1.9.6**; SCA
  workflow #459.

**Follow-ups**
- Opting into config-control on a non-root deploy requires the config mount to be
  writable by uid 1000 (noted in `docker/compose.pg.yaml`).
- Optional later: digest-pin the base images (currently tag-pinned, matching the
  repo convention).

## 2026-06-25 — Signed Android APKs, published on each Release

**Result**
- The Android thin client now builds a **signed** release APK, and CI attaches it
  to every GitHub Release (`.github/workflows/release-apk.yml`, on
  `release: published` + `workflow_dispatch`) — testers download and install it
  straight from the Releases page. Every build uses one stable release key, so
  installs **update in place** across versions without an uninstall.
- Signing is injected into the (gitignored, regenerated) Android Gradle project by
  `prepare-generated-android.mjs` via the new, unit-tested
  `scripts/lib/android-signing.mjs`: keystore from env (CI) or
  `packages/android/keystore.properties` (local). Keyless builds stay unsigned and
  `debug` is unaffected, so CI without the secret and contributors without the key
  still build. PR #468.

**Why / context**
- The `release` build was unsigned (no keystore), so it couldn't be installed or
  updated in place — only `debug` was installable, and side-loaded APKs trip Play
  Protect's unknown-developer block ("Install anyway"). A fixed release key is what
  makes in-place updates work for everyone, so signing had to be durable.
- It's wired through `prepare-generated-android.mjs` rather than hand-edited Gradle
  because `src-tauri/gen/android` is gitignored and regenerated. The injection is
  idempotent and now throws if a future Tauri layout drops an anchor, so it can
  never silently ship unsigned or double-inject. Kotlin DSL gotcha: the injected
  block must use the already-imported `Properties()`, not `java.util.Properties()`
  (the fully-qualified form fails to resolve).
- A version-gate ties the APK version to the release tag (mirrors
  `publish-image.yml`; closes the Android version-drift class from #263), and the
  CI step pins the expected signing-cert SHA-256 so a key swap fails closed.

**Evidence**
- `pnpm typecheck` clean; `pnpm test` 495/495 (6 signing cases). Local end to end:
  a signed universal release APK built (23 MB, arm64); `apksigner` confirms cert
  `CN=Symphony Board` (SHA-256 `a3f2389…d4cb`).
- Keystore master kept in the encrypted secrets store, mirrored as repo secrets
  (`ANDROID_KEYSTORE_BASE64` + store/key passwords + alias) for the CI build.

**Links**
- PR #468 (squash `4365180`). NDK pinned in `packages/android/ndk-version.json`;
  signing + the CI flow documented in `packages/android/README.md`.

**Follow-ups**
- The `release-apk` workflow first runs end to end on the next real release.

## 2026-06-25 — CI security scanning: SCA added; SAST already on (CodeQL default setup)

**Why / context**

**Evidence**

## 2026-06-25 — Reviews recency: `last_comment_at` for long threads (v1.9.4)

**Result**
- Added optional `ReviewThreadDTO.last_comment_at` (contract **4.3.0**): each
  review thread's *true* newest-comment instant, captured at the source
  independent of the comment preview. The Reviews "Recent" sort now prefers it,
  so an actively-updated thread with more comments than the synced preview holds
  no longer sorts as stale. Both sources populate it — GitHub via a
  `comments(last:1)` alias on the review thread, GitLab from the newest
  discussion-note activity — persisted on a new `review_thread.last_comment_at`
  column via additive migration `0008` on **both** drivers (schema `7`→`8`).
  Closes #449 (PR #452).
- Released **v1.9.4** and deployed to the Postgres compose host. Verified live:
  contract `4.3.0`, generator `symphony-board/1.9.4`, `db.schema_version` 8, and
  all 451 review threads carrying `last_comment_at` after the post-deploy full
  sweep re-emitted the contract.

**Why / context**
- Follow-up to #445's "Recent" sort. GitHub fetches only `comments(first:10)`
  (the *oldest* ten), so a >10-comment thread omits its newest comment from the
  preview, and `reviewThreadDisplayTime` fell back to `last_seen_at` — a per-sweep
  observation time, ~identical for every thread in a sweep — burying busy long
  threads. The preview can't carry the true latest instant, so the fix needed
  source + contract + DB work, not a UI tweak. The UI seeds the recency key with
  `last_comment_at` and still lets a newer in-preview comment (an edit) raise it,
  then falls back to the preview, then `last_seen_at`.

**Evidence**
- Test-first red/green: the two new recency tests fail with the `model.ts`
  `last_comment_at` seed reverted, pass with it.
- `pnpm run typecheck && pnpm test` (486 backend) green; UI 324 + build +
  render-smoke clean; `pnpm run test:pg-e2e` (59) and `pnpm run test:pg-compose`
  (asserts `schema_version` 8) green.
- Validated the new GitHub `comments(last:1)` alias against the live GitHub
  GraphQL API before shipping (accepted; returns the newest-comment instant).

**Release-blocker CI fix (PR #454)**
- The `publish-image` "Smoke test images" step waited on the web image with
  `curl .../contract.json | grep -q '"contract_version":'`. Under
  `set -o pipefail`, `grep -q` closes the pipe on its first match and curl's

## 2026-06-24 — GitLab comment activities now link to their note permalink

**Result**
- GitLab comment activities now carry a real URL. The cause, surfaced by the
  real-data demo below: `gitLabProjectEventUrl` had no `comment` case, so every
  GitLab `Note` activity emitted `url: null`, and `DiffNote` / `DiscussionNote`
  (line and threaded review comments) were mis-classified as `repository`
  events. Now any `*Note` event is a `comment` linked to its noteable item,
  anchored at the specific note (`…/-/merge_requests/<iid>#note_<id>`).
- A note event's `target_iid` is the NOTE id, not the parent item iid (the
  parent lives in `event.note.noteable_iid`), so the comment's `targetIid` now
  points at the issue / MR it belongs to. Unresolvable noteables stay unlinked
  rather than guess a location.

**Why / context**
- The gap was deliberate once (a test literally asserted "leave comments
  unlinked") because only `target_iid` was read — and for a note that is the
  note id, so any link would have been wrong. The GitLab events API does carry
  `event.note.{noteable_type,noteable_iid,id}`, which is enough to build a
  correct permalink, so the old decision no longer holds.
- Also retired the public-surface guard marker that hid the sibling repo names:
  the demo aggregates those public repos and renders their data, so guarding the
  names only let the codebase drift from what the demo shows.

**Evidence**
- Test-first: the new `Note` / `DiffNote` / `DiscussionNote` link cases + the
  unresolved-noteable fallback failed before the fix, pass after; full suite
  484/484, typecheck clean.
- On the regenerated demo: GitLab comment activities went from 144 unlinked (plus
  231 mis-bucketed as `repository`) to fully linked.

## 2026-06-23 — Windowed board tail: overlay the gap when display outruns the fetch

**Result**
- Closed #414, the deliberate residual the #413 entry below flagged: on a mobile
  windowed board scope the displayed window could claim a trailing day the
  `/api/range` fetch never requested, so today's items could read as missing in a
  narrow timing window (#417, squash `3eee069`).
- Root cause is a two-clock split: a windowed env is FETCHED on the live `Date.now()`
  clock but DISPLAYED on the contract's `generated_at` (`boardWindowRange`, kept
  byte-identical to its named preset by #407). When a load crosses local midnight — or
  client/server clock skew lands the two on different local days — the `generated_at`
  day can be *later* than the fetched `range_query.to` day. On the landing range
  `activeRange == staticRange`, so `customRange` was `false`, the range overlay never
  fired, and the Activity feed filtered to a window whose last day was never fetched.
- Fix is **option 2 (overlay the gap)**: new pure helper `windowedRangeTailUnfetched`
  in `viewconfig.ts` returns true only when the env is a genuine windowed `/api/range`
  projection AND its displayed tail day outruns the fetched `range_query.to` day

**Why / context**

## 2026-06-22 — Cold-start UX on a slow mobile link, and the bug that hid behind three storage rewrites

**Result**
- Live is usable immediately on cold start: the boot splash holds until the
  view has content, and the cold-start event seed shrank from the buffer cap to
  a small window (a full-cap seed was a ~26MB download dominated by per-event
  `raw` payloads).
- The contract per-attempt request timeout was raised 30s → 60s so a slow /
  relayed link can finish the ~1.5MB gzip download in one attempt instead of
  aborting mid-body and re-fetching the whole payload in a retry loop.
- A per-server, stale-while-revalidate cache of the contract envelope: gzipped
  `localStorage` on web/desktop, and a native-filesystem backend
  (`tauri-plugin-fs`, `AppLocalData`) on the Android shell, because that
  WebView reliably persists neither IndexedDB nor a ~1.5MB `localStorage` value.
- The fix that actually made the cache visible: the blocking "Loading
  contract…" view was gated on the `loading` flag alone, so it covered the
  board the cache had already painted for the entire background revalidation.
  It is now gated on "no content yet" (`loading && !env`) via a unit-tested
  predicate.

**Why / context**
- Symptom vs disease. After Live was made fast, contract-backed pages still sat
  on "Loading contract…". Three storage backends were swapped in turn
  (IndexedDB → localStorage → native FS) chasing "the cache doesn't persist" —
  but the cache was never the problem. On-device evidence proved it read the
  file and set `env` on every cold start; a render overlay simply hid it. The
  storage rewrites were chasing the symptom. The native-FS backend is kept
  because it is a genuinely robust mobile cache (proven on-device), but it was
  never the blocker.
- The real takeaway is a debugging discipline, not a storage choice: when a
  client-side cache "doesn't work" on a device, get device visibility BEFORE
  swapping the mechanism. `adb logcat` filtered to the app uid, `run-as` to
  inspect the app-private data dir, and a few temporary in-code log lines
  replaced three blind iterations and pinpointed a one-line render gate.
- Slow-link nuance: a small live snapshot fit the link and succeeded while the
  larger contract aborted at the 30s ceiling — same link, different payload
  size — which is why Live worked but contract pages didn't until the timeout
  was raised.

**Evidence**
- Server access logs showed the mobile client fetching `contract.json` at the
  gzip size (~1.46MB), confirming gzip negotiation works on the
  `tauri-plugin-http` client; repeated ~30s-spaced fetches exposed the
  timeout-driven retry loop.
- On-device instrumentation logged `fs.exists=true`,
  `fs.read ok bytes=12267417`, and `load supported=true -> PAINT` — i.e. save,
  read, and cache-paint all succeed — while the screen still showed "Loading
  contract…", isolating the render gate as the culprit.
- `run-as` confirmed a complete, valid cache file written under `AppLocalData`;
  a pushed *small* (~30KB) cache file also failed to appear, ruling out a
  payload-size limit and pointing away from storage entirely.
- Validation per change: UI (264 → 272) and root (450 → 457) `node --test`
  suites, `vite build`, render-smoke (0 console errors); Android APK builds
  validated the `tauri-plugin-fs` wiring and a tightly-scoped fs capability.
  Each change was test-first (failing test or recorded waiver).

**Links**
- PRs: #382 (splash hold + seed), #385 (contract request timeout), #386
  (localStorage+gzip cache), #387 (native-FS cache, Android), #389
  (loading-overlay gate — the root-cause fix). Earlier prep: #381.

**Follow-ups**
- The contract is ~15MB decoded / ~1.5MB gzip, dominated by the `activities`
  feed (~56%). A server-side trim would speed up the FIRST cold start; the cache
  only accelerates repeat launches. Deferred this round.
- A mobile-reachable in-app diagnostics surface (cache backend, last save/load
  result, contract source + load time) would make this class of issue
  self-diagnosing without `adb`; the temporary instrumentation proved its value.

## 2026-06-18 — Sub-day TREND buckets for short Repo Analytics windows

**Result**

  sub-day `repo_metrics[].window.bucket` width by day count — 1 day → `2h`, 2 days
  → `4h`, 3 days → `6h` — so every short window tiles into 12 fixed-width points
  aligned to the configured zone's local clock. 4+ day windows are unchanged

**Why / context**
- `today`/`yesterday` previously collapsed to a single bucket: the producer only
  bucketed by `day`/`week`/`month`, and one calendar day is exactly one `day`

## 2026-06-18 — Resilient, compressed contract load (no more stuck "Loading contract…")

**Result**
- The UI's startup contract load is now bounded and self-healing. Each attempt has
  a per-attempt timeout — an `AbortSignal.timeout` ceiling for any client plus a
  plugin-http `connectTimeout` on desktop that bounds *connection setup* without
  aborting an in-progress transfer — and transient failures (network throw,
  abort/timeout, 5xx) retry with exponential backoff. A 4xx, or a 200 whose body is
  not a contract, is definitive and surfaces immediately. The load always settles in
  bounded time, so it can no longer hang forever on `Loading contract…`, and a
  momentary blip self-recovers instead of needing an app restart.
- The UI sidecar now serves the contract gzip'd (`docker/ui-nginx.conf`: `gzip on`
  for `application/json` + JS/CSS), and both desktop crates enable the
  `tauri-plugin-http` `gzip` feature so the Tauri client negotiates and
  transparently decompresses. A large JSON contract drops ~10x on the wire.

**Why / context**
- The contract load was a single `fetch` with no timeout and no retry, served
  uncompressed. Against a remote board over a slow or briefly-unreachable link, the
  one request could stall (UI stuck on `Loading contract…` with no recovery but a
  restart) or fail once and strand the UI empty — reopening "fixed" it only because a
  fresh mount issued a new request. Two user-visible symptoms, one root cause: a
  best-effort one-shot with no recovery path.
- `connectTimeout` + gzip were chosen to compose: gzip shrinks the payload so a
  fixed per-attempt timeout reliably distinguishes a *stalled* connection from a
  *slow-but-progressing* transfer. Gotcha worth keeping: gzip on nginx alone helps
  browsers only — the desktop reqwest (via plugin-http) ships **without** the gzip
  feature, so it neither sends `Accept-Encoding: gzip` nor decompresses; the `gzip`
  feature on `tauri-plugin-http` is what makes server compression reach the desktop.
- Serving the contract is a static read (nginx `alias` over an atomic-rename emit)
  that does **not** contend with the sole-writer sync lock, so an in-progress sync
  was never the cause — the variability tracked the connection/link state, not sync.

**Evidence**
- PR #245 (merge `46e1ca7`), issue #244. Test-first RED→GREEN: 7 new
  `contract.test.ts` resilience cases (retry transient, give up after N, never retry
  a 4xx / non-contract body, honor the per-attempt timeout + `connectTimeout` signal).
- Validation: UI typecheck + 153 UI tests; root typecheck + 257 backend tests; UI
  build + render-smoke; `nginx -t` on the updated config; `cargo fetch` confirmed
  `async-compression` / `compression-codecs` / `compression-core` entered both desktop
  `Cargo.lock`s. CI green (check, ui, coverage, pg, pg_compose, shell, CodeQL).

**Links**
- PR: https://github.com/sympoies/symphony-board/pull/245 (merge `46e1ca7`)
- `packages/ui/src/contract.ts`, `packages/ui/src/runtime.ts`, `docker/ui-nginx.conf`,

## 2026-06-17 — Fix Android status-bar overlap (edge-to-edge insets)

**Result**
- The Android shell now lays the WebView content out below the status and
  navigation bars again. Previously the page drew full-bleed under the status
  bar (clock / icons overlapped the app header).
- Two independent bugs both had to be fixed; either alone left the overlap:
  1. **Inset never delivered (native).** On `targetSdk = 36` Android forces
     edge-to-edge, and the prior fix attached the `OnApplyWindowInsetsListener`
     to the **WebView**. Inside wry's view hierarchy the WebView is not reliably
     in the window-inset dispatch path, so the listener observed a `0`
     status-bar inset. Both earlier attempts — native `setPadding` (`ec628a2`)
     and the CSS safe-area bridge (`c7137d4`) — depended on that same listener,
     so both failed identically. Fix: attach the listener to `window.decorView`,
     which always receives the raw window insets, and fold `displayCutout()`
     into `systemBars()` so notch / punch-hole devices stay clear too.
  2. **Inset delivered but dropped by CSS source order.** Two
     `@media (max-width: 760px)` blocks both set `.app-wide` padding; the later
     `.app, .app-wide { padding: 12px }` won the cascade and clobbered the
     safe-area `calc(...)` block above it, so the mobile header used a flat 12px
     and still slid under the status bar. Fix: move the safe-area insets into the
     winning block and reduce the earlier block to its `overflow-x` guard.
  3. **Scrolled content bled through the transparent bars.** The inset only pads
     the *initial* layout; once the page scrolls, content moves into the
     transparent status / nav-bar regions and shows behind the clock and icons.
     targetSdk 36 can't opt out of edge-to-edge, so the bars can't be made opaque
     by the system. Fix: `body::before` / `body::after` fixed scrims paint the
     bar regions with `var(--bg)` at the safe-area heights (0 in browser/desktop,
     so they're no-ops there), occluding scrolled content behind opaque bars.

**Cleanup**
- Edge-to-edge had been reworked several times, leaving residual CSS. Removed the
  dead `.app` rule (every render is `app app-wide`, and `.app-wide` fully
  overrode its `max-width`/`padding`, so its inset padding never applied),
  dropped the redundant `.app,` selector from the mobile rule, and merged the
  stray `.app-wide { overflow-x }` rule that sat in a separate `@media` block.
  Safe-area insets now live in exactly two `.app-wide` rules (base + mobile) plus
  the two scrims.

**Why / context**
- The generated Android project is gitignored; the native patch lives in
  `packages/android/scripts/prepare-generated-android.mjs`, which runs before
  every `android:dev` / `android:build`. The CSS fix is in
  `packages/ui/src/styles.css`; `runtime.ts` (bridge → CSS vars) was already
  correct and is unchanged.

**Evidence**
- `pnpm --filter @symphony-board/ui test` PASS (134 tests, incl. the safe-area
  bridge test); `pnpm --filter @symphony-board/ui typecheck` PASS.
- On-device verification on a Galaxy S24+ (`SM-S9260`, Android 16 / API 36,
  arm64) via a debug APK + Chrome DevTools probe:

## 2026-06-09 — UI-triggered sync control plane

**Result**
- The board loop is now a Node daemon (`src/cli/sync-daemon.ts`) that owns the

**Why / context**

  header so a blind cross-site POST is rejected; the control port is internal to
  the Compose network. Dry-run keeps `POST` honest — it opens no write
  transaction and emits nothing, so connectivity tests never present unwritten
  state as fresh data. See `docs/DESIGN.md` › "UI-Triggered Sync Control Plane".

## 2026-06-09 — Canonical actor identity for Repo Analytics (contract 2.3.0)

**Result**
- `repo_metrics[].top_actors[]` now groups by a canonical `actor_key` instead of
  the raw display string, so one human is one row. Each row gained `actor_key`,
  `display_name`, and optional `aliases`; `actor` stays as a back-compat display
  field. Additive minor bump `2.2.1 → 2.3.0`.
- New `deriveActorKey` helper (`src/model/actor.ts`), additive DB migration
  `0003` (`activity.actor_key`, backfilled on upsert), and source normalizers
  that populate the key — including #93's review/approval activities.

**Why / context**
- Key precedence is **provider username → hashed commit email → normalized
  name**, deliberately *not* the issue's suggested email-first ordering: a
  GitHub user's issues carry a login but their commits carry an email, so
  email-first would re-split the same human — the exact duplication we were
  fixing. Username-first merges account-linked commits with issues; email is the
  fallback that collapses account-less commit author-name variants. The email is
  SHA-256-hashed, so a raw address never enters the canonical store or the
  contract.
- Live data bore this out: on GitLab repos, commit name-variants like
  `Dev C` / `Dev C (alt)` and `Dev A` / `Dev A (alt)` collapsed into one
  email-keyed identity with aliases. The username and email facets of one GitLab
  human remain separate rows (events carry a username, commits carry an email,
  with no provider cross-link) — a documented follow-up, not the bug this fixed.

**Evidence**

## 2026-06-08 — Root-caused & killed the raw-NUL-in-source bug class (#41)

**Result** — #41 squash-merged (`7310de8`), closing the follow-up flagged in the
#32 entry below:
- Three composite Map-key builders joined fields with a NUL (`\0`) separator, and
  some were committed with a **literal `0x00` byte** instead of the source escape:
  `src/model/edges.ts` `keyOf` (2), `packages/ui/src/model.ts` `relatedItems` (1);
  `src/sources/gitlab.ts` `indexKey` had been the `\0`-escape form (fixed in #33).
- All three now build their key with `JSON.stringify([...])` — the same
  collision-proof convention as the UI model's `repoKey` — so there is **no
  separator byte to mis-save** at all. (`relatedItems` was independently rewritten
  in #40 to key by `ref`, so that site landed NUL-free via the rebase.)
- New `test/no-nul-bytes.test.ts` enumerates `git ls-files` and fails on any
  `0x00` byte; it runs in CI's `check` job and the pre-push hook, and was
  negative-tested (fails + names the file when a NUL is staged).

**Why / root cause**
- **Authoring-time, not a formatter.** The repo has **no formatter** (no
  prettier/biome/eslint; lefthook only runs typecheck + test), so nothing
  rewrites files on commit — whatever agent wrote the file emitted a raw NUL from
  an intended `\0`, and it was committed verbatim. Claude Code's own Edit/Write
  round-trip `\0` correctly (the #33 fix stayed clean), so the culprit is a
  different write path; commits are all signed `dev-a`, so history can't name
  the tool.
- **Why it survived for so long:** a NUL inside a JS/TS string is *valid* — `tsc`,
  the runtime, and behaviour tests (which assert results, not key bytes) never
  flag it. The only symptoms are `file`/`grep` seeing the file as binary and a
  `git diff` "Bin …" when the byte is within git's ~8 KB binary-sniff window.

**Evidence** — `pnpm run typecheck` + `pnpm test` (40 pass incl. the new guard),
UI build + `… run test` (14 pass), and a full-tree NUL scan → zero. The fix is
behaviour-preserving: every changed key is an in-memory dedup Map key, never
persisted to the DB or contract.

  `relatedMergeRequests` (the existing `closes` source). Live introspection
  confirmed this, so the chosen "both relates + mentions" is realized by parsing
  human-readable **system notes** instead (locale/version dependent, brittle by
  nature). Notes are fetched **per item** (an N+1, like `relatedMergeRequests`):
  inlining `notes(first:N)` in the bulk page would exceed GitLab's query-complexity
  limit. A mention that duplicates a `closes` edge (the closing MR) is skipped,
  mirroring GitHub's `willCloseTarget`; the symmetric `relates` link is
  canonicalized so reconcile merges the two reports into one edge. `normalize`
  stays pure — `fetch` resolves iid → global id (via the sweep's items) and
  attaches the endpoints to the payload.
- **The UI build can't run the workspace `prepare` in Docker** — lefthook's
  `install` needs a git repo the image hasn't got. `pnpm install --ignore-scripts
  && pnpm rebuild esbuild` skips `prepare` while still building Vite's one
  allowlisted dependency.
- **Sole-writer holds end to end**: the sidecar `depends_on` the board
  healthcheck (`condition: service_healthy`), so the reader never serves before
  the writer has produced a contract.

**Why / context**
- **pnpm 11 build-script gotcha (the keeper).** lefthook ships a postinstall;
  pnpm 11 blocks unapproved build scripts *and fails the install* (`exit 1`,
  via the pre-run deps-status check) — this would have broken CI's
  `pnpm install --frozen-lockfile`. The approval/ignore config has **moved to
  `pnpm-workspace.yaml`** (`allowBuilds: { <dep>: true|false }`); the
  package.json `pnpm` field (incl. `ignoredBuiltDependencies`) is **no longer
  honored** in 11.4.0 — verified by a fresh reinstall still erroring with the
  field set. Fix: `allowBuilds: { lefthook: false }` (acknowledge, don't run the
  dep's postinstall) + wire hooks via our own `prepare` (keeps the repo's
  minimal-trust, no-dependency-build-scripts posture).
- Chose **pre-push** over pre-commit (whole-project gate, no per-commit
  friction); **no commit-msg hook** (commit format is owned by the commit flow);
  the live `--dry-run` smoke is deliberately **not** gated (needs a token /
  possibly VPN-bound GitLab) — same reason CI skips it.

**Evidence**
