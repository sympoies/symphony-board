Read-only · GitHub + GitLab

One board for every provider's work items

Symphony Board coordinates issues, pull requests, merge requests, activity, and typed relationships from configured GitHub and GitLab projects into a canonical store, emits a versioned JSON contract, and serves a read-only web UI from it.

▶ Launch live demo View on GitHub

What it does

The product surface is the contract plus the UI. The database is an implementation store, not the consumer API.

◆Cross-provider sources

GitHub and GitLab GraphQL sources fetch issues and change requests; REST activity surfaces add commits, comments, and repository/project events.

◆Canonical store

Raw provider payloads normalize into a provider-agnostic canonical database — SQLite by default, Postgres by opt-in compose.

◆Versioned contract

A validated, versioned JSON contract (major v4) is the consumer API. The canonical schema is never exposed directly.

◆Read-only UI

A 7-column Board, relationship Graph, Activity feed, Commits log, Review-thread inbox, and Repo Analytics — all over one shared date-range control.

◆Apps

Thin macOS (Tauri) and Android clients over a server HTTP surface, plus a fully self-contained standalone macOS app with a bundled backend.

◆One writer per store

A writer lease guarantees a single producer per store. The UI is read-only toward providers — it never writes issues, labels, or PR/MRs back.

How it works

Three separate layers — raw store, canonical DB, versioned contract — keep the data model honest end to end.

providers ──fetch──▶ [1] raw store      opaque provider payloads
                          │ normalize (pure)
                          ▼
                     [2] canonical DB   item / edge / activity / label / sync_*
                          │ buildContract (pure)
                          ▼
                     [3] contract       versioned JSON ──▶ UI / consumers
ℹ️
About the live demo. It runs entirely in your browser against a bundled sample contract, so no provider data leaves your machine. Backend-only features — live refresh, time-window queries, and the sync / config control plane — require a running server and are inactive in this static demo.
