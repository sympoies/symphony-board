## 2026-07-02 — Live capabilities diagnostics

**Result**
- Added a read-side `GET /api/capabilities` surface for board route availability
  and Live read status (`unsupported` / `unreachable` / `empty` / `ready`).
- Settings now explains Live read availability, the latest retained Live
  sequence/time, optional non-secret webhook setup hints, and allowlist count
  without adding a second Live URL setting.
- Docker wiring keeps the browser on the existing Server URL for
  `/contract.json`, `/api/range`, `/api/capabilities`, and `/api/live*`; public
  webhook ingress remains deployment-owned and is still not proxied by the UI
  sidecar.

**Why / context**
- Hosted and thin-client users need to diagnose "Live unavailable" versus "Live
  receiver reachable but no events yet" from the same read-side base URL they
  already configure. The capability document is intentionally operational
  metadata, not a contract bump: it may report route booleans, latest Live
  sequence/time, provider names, subscribed-event hints, and allowlist counts,
  but never credential material.

## 2026-07-01 — Clear the Code Scanning backlog + drain the dependabot queue

**Result**
- Took open Code Scanning alerts from 10 → **0** across two releases, and the
  open PR queue to **0**.
- **Board image (v1.12.7, #541):** dropped build-only tooling from the runtime
  layer — the base image's bundled `npm` (its own vulnerable `undici` + `tar`)
  and the corepack `pnpm` cache — and bumped the pnpm pin 11.7.0 → 11.9.0. That
  cleared all 10 board-image alerts (bundled-`undici` ×4, `tar`, corepack-`undici`
  ×4, the `pnpm` advisory).
- **Web image (v1.12.8, #542):** upgraded `libexpat` 2.8.1-r0 → 2.8.2-r0 in the
  nginx stage, clearing the 13-CVE cluster (CVE-2026-50219 / -561xx / -564xx) a
  fresh nginx-alpine pull had surfaced.
- **Dependency queue:** consolidated 8 open npm dependabot PRs into one coherent
  lockfile update (#543) and merged 3 GitHub Actions bumps (#530/#531/#533).
  Shipped with the libexpat fix in v1.12.8.

**Why / context**
- Every Trivy *image* finding lived in base-image or build tooling the runtime
  never executes — the container entrypoint only runs `node`, so the fix was to
  stop *shipping* the tooling, not to patch an app dependency. The board image
  is single-stage and had left `npm` + the corepack cache in the final layer;
  removing them in the same install `RUN` keeps them out of the image entirely
  (and shrinks it). The web image's `libexpat` is a transitive OS package; the
  UI sidecar only serves static files (never parses XML), so exposure was
  negligible, but a pinned `apk add --upgrade "libexpat>=2.8.2-r0"` (fails
  closed) keeps the scan clean.
- The React bump *had* to be consolidated: dependabot split `react` (#537) from
  `react-dom` (#538), so each failed CI alone with a React 18/19 internals
  mismatch (`ReactCurrentBatchConfig` undefined). Bumped as a set — react +
  react-dom 19, matching `@types`, `@vitejs/plugin-react` 6, `vite` 8 — the app
  needed **no code changes**.
- Image-scan alerts only reconcile after a new `:latest` is published *and* the
  `trivy-image` job re-runs (it is `schedule`/`workflow_dispatch`-only, scanning
  the published tag). Merging a fix never clears the alert on its own — each fix
  needed a release plus a manual `security.yml` dispatch to confirm.

**Evidence**
- Local Trivy scans of the rebuilt images: board = 0 vulns (npm / corepack /
  undici / tar gone, `node` + `postgres` intact); web = 0 vulns (`libexpat`
  2.8.2-r0).
- React 19 batch: root 574/574, UI 359/359, UI + standalone `vite` 8 builds,
  and both render-smokes PASS (0 console errors / 0 uncaught exceptions — the
  exact net for the crash class); CI 12/12 required green.
- Alerts: board #11–20 → fixed after v1.12.7 + rescan; web `libexpat` #21–33 →
  fixed after v1.12.8 + rescan. Final: **0 open alerts, 0 open PRs.**
- Both releases deployed to the self-hosted production stack via its
  release-env bump PR; all board + web containers verified on the new tag and
  healthy.
