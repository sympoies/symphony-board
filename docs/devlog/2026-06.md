## 2026-06-10 — Docker startup progress logs

**Result**
- Added provider fetch progress logs to the always-on `board` container so a
  slow initial full sweep no longer looks silent while `api` and `web` wait on
  `board` health.
- GitHub source logs each configured project through GraphQL issue/PR fetches
  and REST activity fetches.
- GitLab source logs each configured project through bulk issue/MR fetches,
  per-item resolve progress every 25 items, and REST activity fetches.

**Why / context**
- A cold Docker start can spend several minutes on the first full sweep before
  `data/contract.json` is fresh enough for the healthcheck. Before this change,
  the log only showed source-level `syncing...` and final status, so a long
  source such as Gamania GitLab looked stuck.
- The new logs preserve the current sole-writer model and do not change sync,
  normalization, DB, or contract behavior; they only expose progress already
  happening inside provider fetches.

## 2026-06-09 — UI-triggered sync control plane

**Result**
- The board loop is now a Node daemon (`src/cli/sync-daemon.ts`) that owns the
  cadence, a single run lock, and a writer-owned HTTP control surface
  (`POST /api/sync-runs`, `GET /api/sync-runs/current|last`, `/api/sync-control`).
- The UI gained a **Sync** action in the Header (incremental, all sources) and an
  advanced Settings section (full sweep, dry-run, source scope). A successful,
  non-dry run reloads `contract.json` (and the active range) in place, preserving
  route/search/filters/range.
- `sync` and `emit` were refactored onto a shared runner (`src/sync-runner.ts` +
  `src/contract/emit.ts`); Docker `loop` delegates to the daemon and nginx routes
  the sync-control paths to `board` while `/api/range` stays on the read-only
  `api` sidecar.

**Why / context**
- Browser reload only re-reads the last emitted contract; it never asks
  providers for new data. A real "Sync now" can't be a plain UI button calling
  the read-only API, because `board` must stay the only DB writer. The fix is a
  narrow writer-owned control plane: one process, one lock, so a manual run and a
  scheduled tick never overlap on SQLite. Manual runs are opt-in
  (`SYNC_CONTROL_ENABLED`) and guarded by a same-origin `X-Symphony-Sync-Control`
  header so a blind cross-site POST is rejected; the control port is internal to
  the Compose network. Dry-run keeps `POST` honest — it opens no write
  transaction and emits nothing, so connectivity tests never present unwritten
  state as fresh data. See `docs/DESIGN.md` › "UI-Triggered Sync Control Plane".

**Evidence**
- `pnpm run typecheck` — clean.
- `pnpm test` — 115/115 backend tests pass (adds `test/sync-runner.test.ts`,
  `test/sync-daemon.test.ts`: emit gating, the single lock, re-entrancy 409,

## 2026-06-09 — Canonical actor identity for Repo Analytics (contract 2.3.0)

**Result**
- `repo_metrics[].top_actors[]` now groups by a canonical `actor_key` instead of
  the raw display string, so one human is one row. Each row gained `actor_key`,
  `display_name`, and optional `aliases`; `actor` stays as a back-compat display
  field. Additive minor bump `2.2.1 → 2.3.0`.
- New `deriveActorKey` helper (`src/model/actor.ts`), additive DB migration
  `0003` (`activity.actor_key`, backfilled on upsert), and source normalizers
  that populate the key — including #93's review/approval activities.

**Why / context**
- Key precedence is **provider username → hashed commit email → normalized
  name**, deliberately *not* the issue's suggested email-first ordering: a
  GitHub user's issues carry a login but their commits carry an email, so
  email-first would re-split the same human — the exact duplication we were
  fixing. Username-first merges account-linked commits with issues; email is the
  fallback that collapses account-less commit author-name variants. The email is
  SHA-256-hashed, so a raw address never enters the canonical store or the
  contract.
- Live data bore this out: on GitLab repos, commit name-variants like
  `Wayne XU(許恒彬)`/`Wayne` and `Terry LIN 林品澄`/`Terry LIN` collapsed into one
  email-keyed identity with aliases. The username and email facets of one GitLab
  human remain separate rows (events carry a username, commits carry an email,
  with no provider cross-link) — a documented follow-up, not the bug this fixed.

**Evidence**

## 2026-06-08 — Root-caused & killed the raw-NUL-in-source bug class (#41)

**Result** — #41 squash-merged (`7310de8`), closing the follow-up flagged in the
#32 entry below:
- Three composite Map-key builders joined fields with a NUL (`\0`) separator, and
  some were committed with a **literal `0x00` byte** instead of the source escape:
  `src/model/edges.ts` `keyOf` (2), `packages/ui/src/model.ts` `relatedItems` (1);
  `src/sources/gitlab.ts` `indexKey` had been the `\0`-escape form (fixed in #33).
- All three now build their key with `JSON.stringify([...])` — the same
  collision-proof convention as the UI model's `repoKey` — so there is **no
  separator byte to mis-save** at all. (`relatedItems` was independently rewritten
  in #40 to key by `ref`, so that site landed NUL-free via the rebase.)
- New `test/no-nul-bytes.test.ts` enumerates `git ls-files` and fails on any
  `0x00` byte; it runs in CI's `check` job and the pre-push hook, and was
  negative-tested (fails + names the file when a NUL is staged).

**Why / root cause**
- **Authoring-time, not a formatter.** The repo has **no formatter** (no
  prettier/biome/eslint; lefthook only runs typecheck + test), so nothing
  rewrites files on commit — whatever agent wrote the file emitted a raw NUL from
  an intended `\0`, and it was committed verbatim. Claude Code's own Edit/Write
  round-trip `\0` correctly (the #33 fix stayed clean), so the culprit is a
  different write path; commits are all signed `graysurf`, so history can't name
  the tool.
- **Why it survived for so long:** a NUL inside a JS/TS string is *valid* — `tsc`,
  the runtime, and behaviour tests (which assert results, not key bytes) never
  flag it. The only symptoms are `file`/`grep` seeing the file as binary and a
  `git diff` "Bin …" when the byte is within git's ~8 KB binary-sniff window.

**Evidence** — `pnpm run typecheck` + `pnpm test` (40 pass incl. the new guard),
UI build + `… run test` (14 pass), and a full-tree NUL scan → zero. The fix is
behaviour-preserving: every changed key is an in-memory dedup Map key, never
persisted to the DB or contract.

**Why / context**
- **pnpm 11 build-script gotcha (the keeper).** lefthook ships a postinstall;
  pnpm 11 blocks unapproved build scripts *and fails the install* (`exit 1`,
  via the pre-run deps-status check) — this would have broken CI's
  `pnpm install --frozen-lockfile`. The approval/ignore config has **moved to
  `pnpm-workspace.yaml`** (`allowBuilds: { <dep>: true|false }`); the
  package.json `pnpm` field (incl. `ignoredBuiltDependencies`) is **no longer
  honored** in 11.4.0 — verified by a fresh reinstall still erroring with the
  field set. Fix: `allowBuilds: { lefthook: false }` (acknowledge, don't run the
  dep's postinstall) + wire hooks via our own `prepare` (keeps the repo's
  minimal-trust, no-dependency-build-scripts posture).
- Chose **pre-push** over pre-commit (whole-project gate, no per-commit
  friction); **no commit-msg hook** (commit format is owned by the commit flow);
  the live `--dry-run` smoke is deliberately **not** gated (needs a token /
  possibly VPN-bound GitLab) — same reason CI skips it.
