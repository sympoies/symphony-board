## 2026-06-08 — Root-caused & killed the raw-NUL-in-source bug class (#41)

**Result** — #41 squash-merged (`7310de8`), closing the follow-up flagged in the
#32 entry below:
- Three composite Map-key builders joined fields with a NUL (`\0`) separator, and
  some were committed with a **literal `0x00` byte** instead of the source escape:
  `src/model/edges.ts` `keyOf` (2), `packages/ui/src/model.ts` `relatedItems` (1);
  `src/sources/gitlab.ts` `indexKey` had been the `\0`-escape form (fixed in #33).
- All three now build their key with `JSON.stringify([...])` — the same
  collision-proof convention as the UI model's `repoKey` — so there is **no
  separator byte to mis-save** at all. (`relatedItems` was independently rewritten
  in #40 to key by `ref`, so that site landed NUL-free via the rebase.)
- New `test/no-nul-bytes.test.ts` enumerates `git ls-files` and fails on any
  `0x00` byte; it runs in CI's `check` job and the pre-push hook, and was
  negative-tested (fails + names the file when a NUL is staged).

**Why / root cause**
- **Authoring-time, not a formatter.** The repo has **no formatter** (no
  prettier/biome/eslint; lefthook only runs typecheck + test), so nothing
  rewrites files on commit — whatever agent wrote the file emitted a raw NUL from
  an intended `\0`, and it was committed verbatim. Claude Code's own Edit/Write
  round-trip `\0` correctly (the #33 fix stayed clean), so the culprit is a
  different write path; commits are all signed `graysurf`, so history can't name
  the tool.
- **Why it survived for so long:** a NUL inside a JS/TS string is *valid* — `tsc`,
  the runtime, and behaviour tests (which assert results, not key bytes) never
  flag it. The only symptoms are `file`/`grep` seeing the file as binary and a
  `git diff` "Bin …" when the byte is within git's ~8 KB binary-sniff window.

**Evidence** — `pnpm run typecheck` + `pnpm test` (40 pass incl. the new guard),
UI build + `… run test` (14 pass), and a full-tree NUL scan → zero. The fix is
behaviour-preserving: every changed key is an in-memory dedup Map key, never
persisted to the DB or contract.

**Links** — PR #41 (`7310de8`); prior occurrence #33; superseding rewrite #40.

## 2026-06-07 — merge_state shown only while a PR/MR is open (#33)

**Result** — #33 squash-merged (`ad2081b`):
- Board cards stopped showing a meaningless merge badge on closed items. A merged
  GitHub PR was rendering `merge: unknown` and a merged GitLab MR `merge:
  mergeable`, right beside the `merged` lifecycle badge.
- `normalize` now gates `mergeState` on the open lifecycle state for both
  sources; a non-open item carries `mergeState: null`, which the already-nullable
  contract field and the UI's conditional badge drop cleanly. No contract bump,
  no schema change; CI state left untouched.
- Same PR, separate commit: removed two embedded raw **NUL (0x00) bytes** in the
  `gitlab.ts` `indexKey` separators that made git/grep/diff treat the whole file
  as binary. Replaced with the `\0` escape — byte-identical at runtime, but the
  file (and this PR's diff) is plain text again.

**Why / context**
- **Mergeability is an open-PR concept.** GitHub stops computing `mergeable`
  after merge/close and returns `UNKNOWN` (→ `"unknown"`); GitLab's
  `detailedMergeStatus` returns `NOT_OPEN` (→ `"mergeable"`). Both are stale noise
  next to the lifecycle badge, so they are dropped rather than re-mapped. The
  legitimate transient `unknown` while a PR is *still open* is preserved.
- **Fixed in the pure `normalize` layer**, so it replays against stored raw with
  no re-fetch: existing rows repopulate on the next sweep / re-normalize and the
  loop daemon converges.
- **The NUL bytes were a latent trap**, not part of this bug — but they blocked a
  reviewable diff of the `gitlab.ts` change, so they were fixed first in their own
  commit.

**Evidence**
- `pnpm run typecheck` + `pnpm test` → 39 pass (4 new `normalize` tests: GitHub &
  GitLab × open/merged). `--dry-run` sync waived (no `GITHUB_TOKEN`/VPN in shell;
  normalize is pure and unit-tested). CI on #33: `check` ×2 + `ui` ×2 +
  CodeQL/Analyze green.

**Links** — PR #33 (squash `ad2081b`); reported from a board-card screenshot of
the merged #28.

**Follow-ups** — a re-normalize (or the next daemon sweep) is what actually clears
the stale `merge_state` already stored for previously-merged items.

## 2026-06-07 — Settings page: client-side per-repo display filter (#23)

**Result** — #23 shipped via squash PR #26 (`9d24b50`):
- A **Settings** page (`#/settings` + a Settings tab, filling the slot freed by
  removing the Debug page #25): per-repo checkboxes grouped by source — each
  source's read-only sync health beside its repos — with select-all/none and
  item counts.
- The choice persists in `localStorage` and is applied as a **global pre-filter
  before** the transient facet toggles: it narrows items **and** their edges into
  a `visibleEnv` that feeds the Board, Graph, and stats, so unchecking a repo
  removes its items and edges everywhere.

**Why / context**
- **Display-only, client-side — deliberately NOT editing the daemon's sync
  config.** Changing what the daemon syncs would need a write API behind the
  static nginx sidecar and break the read-only-consumer architecture; the daemon
  stays the sole writer and keeps syncing every source. No backend, no contract
  bump.
- **A pre-filter, not another facet.** It sits *before* the existing transient
  `source`/`state`/`kind` toggles and is the only view state that persists.
  Statuses are derived over the post-pre-filter `visibleEnv`, so a hidden repo's
  edges stop influencing Trailing / In-Progress everywhere — not just hidden from
  the column placement.
- **Store the HIDDEN set, not the visible set.** Tracking hidden keys keeps
  "everything visible" the default as new repos appear in later syncs.
- **`repoKey` is a JSON tuple** (pre-merge review repair, squashed in): a
  collision-proof composite key with no separator assumption — it doubles as the
  `localStorage` unit.

## 2026-06-07 — Contract pipeline hardening + the UI lands

**Result** — one branch (`feat/contract-pipeline-hardening`) closed every
remaining v1 deferral except raw history, and shipped the first UI:
- **Contract validator** (`9325123`): a dependency-free JSON-Schema-subset
  validator; `emit` now refuses to write an invalid contract. No ajv — keeps the
  backend's zero-third-party-dep posture.
- **Edge soft-delete** (`6b0b3dd`): a full+complete sweep tombstones unseen
  intra-source edges, symmetric to items.
- **Incremental loop cadence** (`f6bf2f7`): the daemon runs `--incremental` most
  iterations and a full sweep every `FULL_EVERY` (default 12 ≈ hourly), so the
  soft-delete stays correct while routine runs stay cheap.
- **`@symphony-board/contract`** (`5636dc2`): LAYER 3 extracted into a workspace
  package (pnpm workspace); the three-layer boundary is now structural.
- **`packages/ui`** (`15a84a1`): a read-only board (Vite + React) consuming the
  contract — source health, filters/grouping, label chips, and the issue↔PR/MR
  relationship view by lifecycle.

**Why / context**
- **The contract package is type-only at runtime — on purpose.** Node refuses to
  strip types from files under `node_modules`, so a runtime *value* imported from
  a workspace package would have forced a build step onto the backend. Keeping
  the producer constants (`CONTRACT_VERSION`, `GENERATOR`) and the validator on
  the producer side, and exporting only types + the JSON schema from the package,
  means the backend's `import type`s erase and it never resolves the package at
  runtime — so it still ships with no build and no `node_modules` in the Docker
  image. Proven by running build+validate with the workspace link removed.
- **The UI is the deliberate home for a build step.** DESIGN anticipated "heavy
  deps + a build step" for the UI; Vite/esbuild is exactly that. esbuild's
  build-script is the single trusted exception in the workspace `allowBuilds`
  allowlist, scoped to the UI's tooling — the backend stays minimal-trust.
- **The engine is a forwarder; the source owns the watermark logic.** A unit test
  pinned this (full → source nulls `since`; incremental → stops at the first
  record older than the watermark), plus a fake-Source engine test for the
  "incremental/partial never tombstone" gate.

**Evidence**
